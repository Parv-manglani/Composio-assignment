The agent — what actually did the research

A 10-way parallel research pipeline, orchestrated by Claude Code

Composio's own MCP tools were used to check real toolkit coverage before any web research happened — the same instinct a human researcher would have (check what already exists before assuming it needs to be built).

1

Split by category

The 100-app list was split into its 10 given categories (10 apps each) so each research thread stays scoped and comparable.

2

10 parallel subagents

Each subagent got the same strict JSON schema and instructions, and ran independently with its own WebSearch + WebFetch + Composio MCP access.

3

Composio-toolkit check first

Before researching an app's raw docs, each subagent checked composio.dev/toolkits/<slug> and cross-referenced Composio's own catalog, so "buildable" also means "does this already half-exist."

4

Structured extraction

Each app produced one JSON object: category, auth, self-serve tier, API surface, Composio/MCP status, verdict, blocker, and evidence URLs — no free text allowed to slip through ungrounded.

Where a human was needed

  • Resolving contradictions between subagents. A few apps (e.g. Twilio's MCP server, GitHub's exact tool count) got confident-sounding but slightly wrong answers — caught only by an independent re-check against the primary source, not by the research agent grading itself.
  • Judgment calls on ambiguous gating. Whether "a free trial that requires 3 seats minimum" counts as self-serve, or whether a sandbox-only API counts as "ready," needed a human-set convention applied consistently across categories.
  • Deciding when "not found" is a real finding. For long-tail apps (Paygent Connect, Waterfall.io, fanbasis), a human had to decide the agents had genuinely exhausted reasonable search effort rather than just given up early.
  • The verification pass itself. Every correction on this page came from a second, independent look — see below.

Reusable pipeline code: agent/research.mjs (Composio toolkit check + Claude with the web_search tool, same schema as above) and agent/verify.mjs (automated evidence-link liveness check). See the repo README for how to run them with your own API keys.

Automated link-liveness check — 25 random apps

The agent/verify.mjs script (no API key needed) HTTP-fetches every evidence URL for a random sample and flags non-2xx responses. Re-running it live on this sample found that most flagged links were 403s from bot-protected sites (Otter.ai, PitchBook) — false positives for "broken," not actual dead links — while one (a guessed Composio toolkit slug for SE Ranking) was a genuine 404 that's consistent with that app correctly being marked as having no Composio toolkit. This is disclosed rather than smoothed over: link-liveness is a coarse, fast signal, not proof.

Where the agent got it wrong, or gave up

  • Reducto: first pass claimed a working Composio toolkit existed; the actual docs page says "Composio Managed App Available? N/A." Corrected after verification.
  • Twilio: first pass described Twilio's MCP server as the thing that executes API calls; it's actually a read-only docs/spec search tool, and a separate community project does the execution. Corrected.
  • GitHub: Composio tool count was off by about 5% (846 vs. an estimated ~893). Corrected to the exact number from the toolkit page.
  • Pinterest Ads: one claim (Standard-access review requirement) could not be re-confirmed within the verification budget because the source URL 404'd on re-fetch. Flagged as unverified in the dataset rather than silently trusted.
  • Paygent Connect, Waterfall.io, NotebookLM (consumer), fanbasis: no usable public self-serve developer docs were found for these at all — reported as "blocked / no public docs found," which is a legitimate finding, not a research failure.
